# billsim

`billsim` is a portable, headless billiards simulation core for Railbird. It
reimplements the narrow Pooltool 0.4.4 surface used by Shot Lab, rather than the
rendering, game, and Panda3D portions of Pooltool.

The core uses SI units. Positions are physics-table coordinates in meters:
`x` spans the short dimension and `y` spans the long dimension. Image
homography and UI coordinate conversion remain consumer concerns.

## Implemented surface

- Pooltool-compatible cue impact (`speed`, `phi`, `theta`, `a`, `b`), including
  squirt, follow/draw, and side spin;
- sliding, rolling, spinning, and stationary cloth motion;
- analytic event detection and frictional/inelastic ball-ball response;
- the exact Pooltool six-pocket table construction: 18 linear rail faces, 12
  circular jaws, six named pockets, and canonical pocket resolution;
- Mathavan 2010 linear and circular cushion response with Pooltool's defaults;
- dense trajectories, event history, final state, and potted-ball reporting;
- jaw-aware geometric potting, cut angle, required precision, obstruction
  reporting, and simulation-refined spin/throw compensation;
- a position-play search (`suggest_position_shot`): sweep speed/spin for
  strikes that pot a target ball and leave the cue ball inside a target
  polygon, ranked by speed tolerance and dwell, restricted to contact-clean
  shots (the single cue -> target hit is the only ball-ball contact);
- a serialization-first FFI contract exported through PyO3 and UniFFI; and
- an optional Python adapter implementing Railbird's existing `ShotSimulator`
  protocol and pot-aim result shape.

The differential tests in
[`tests/pooltool_044_conformance.rs`](tests/pooltool_044_conformance.rs) use
fixtures generated by Pooltool 0.4.4. Representative rail, pocket, and
multi-collision shots agree at tight floating-point tolerances.

## Rust

```sh
cargo test --all-features
cargo clippy --all-targets --all-features -- -D warnings
```

The primary APIs are `simulate`, `compute_pot_aim`, and
`suggest_position_shot`. `simulate_json`, `compute_pot_aim_json`, and
`suggest_position_shot_json` expose the same types as JSON for language
boundaries.
Input table and options objects support omitted fields through the documented
Rust defaults. See [`docs/wire-format.md`](docs/wire-format.md) for JSON
examples.

Run the representative two-ball throughput harness with:

```sh
cargo run --release --example throughput -- 10000
```

## Python and Railbird

Build a wheel with Maturin:

```sh
maturin build --release
```

The extension exports `billsim.simulate_json` and
`billsim.compute_pot_aim_json`. The optional
`billsim.railbird.RustShotSimulator` maps Railbird plane coordinates and
datatypes onto the Rust contract without changing Railbird's caller-facing
interface.

## iOS and Android

The `uniffi-bindings` feature exports the JSON boundary to Swift and Kotlin.
See [`docs/mobile.md`](docs/mobile.md) for native builds, generated bindings,
and the intended React Native/Nitro integration seam.

## Remaining optimizations and extensions

- collision-time caching for large ball sets (the current event loop is
  behavior-complete but recomputes candidates after each event);
- a dedicated React Native package around Railbird's existing Nitro module
  conventions; and
- an optional WebAssembly wrapper.

## WebAssembly

`wasm-bindings/` is a thin `wasm-bindgen` wrapper crate (separate because
billsim forbids `unsafe_code`, which the `#[wasm_bindgen]` expansion
violates). It exports the three `*_json` functions for browsers and
React Native web:

```sh
cd wasm-bindings
wasm-pack build --target web --release
```

The full default position search (108 cells) runs in roughly 36 ms under
Node's WASM runtime.
